# AI Career Mentor — FRONTEND Build Prompt (Free-Tier Stack)

Standalone prompt for building only the React frontend. Build this against the API
Contract in Section 3 — you do not need the backend running; mock it with MSW (Mock
Service Worker) so frontend work can proceed in parallel with backend work.

**Stack:** React 18 + Vite + TypeScript + Tailwind CSS + Cloudinary (direct upload)

---

## 0. How to Run This Without Vibe Coding

"Vibe coding" here means: running an AI tool in full-auto mode, accepting whatever it
generates without reading it, and treating "tests pass" as proof the code is correct. This
prompt is written to make that hard to do by accident — but the rules only work if you (the
human) actually follow the process, not just paste the prompt and walk away:

- **Don't use "accept all" / full-auto mode.** Review each diff before it's applied,
  especially anything touching auth, token storage, or the Cloudinary upload validation.
- **Build in the increments listed in "Build in this order" below — don't ask the agent to
  build the whole app in one pass.** If you find yourself about to say "just do all of it,"
  that's the failure mode this is designed to prevent.
- **At every point marked `STOP` below, actually stop.** Read what was generated, ask the
  agent to explain any part you don't understand, and only say "continue" once you do.
- **Run the tests yourself and read what they assert.** A passing suite written by the same
  agent that wrote the code it's testing is a starting signal, not a guarantee.
- **Treat any stub, placeholder, or "simplified for now" comment as an open task**, not as
  done — the prompt requires the agent to flag these explicitly so they don't get lost.

## 1. MASTER PROMPT

````
You are a senior frontend engineer. Build the production frontend for "AI Career Mentor"
— a React + Vite + TypeScript + Tailwind app that lets a user upload a resume, see AI-
generated skill-gap analysis, career path recommendations, job matches, interview practice
questions, a learning roadmap, and chat with an AI career coach.

## Scope
Frontend ONLY. You are not building the backend. The backend is a Django + DRF REST API
(JWT-authenticated) described in the API Contract below — code against that contract
exactly, including request/response shapes, status codes, and error format. If the contract
is ambiguous, make the most standard REST assumption and note it in a comment.

## Non-negotiable standards
- TypeScript throughout, strict mode on. No `any` except at well-justified boundaries.
- Every page has loading, empty, and error states — not just the happy path.
- All forms validated client-side with zod schemas before submission.
- No secrets in the bundle other than public, designed-to-be-public values (Cloudinary
  cloud name + unsigned upload preset, public API base URL).
- Accessible: semantic HTML, labeled form fields, keyboard-navigable, sufficient color
  contrast (this is a Tailwind app — use the design tokens from the design guidance you
  load, don't invent ad hoc colors).

## Engineering discipline — read this before writing any code
This is not a one-shot generation task. Follow these rules for every step below:
- **Work one unit at a time** — one hook, one page, one component — not "the whole resume
  feature." After each unit, stop and write a short summary: what you built, why you chose
  this approach over alternatives you considered, and what the developer should manually
  check before moving on.
- **Explain non-trivial logic before you write it.** The token-refresh interceptor and the
  Cloudinary upload flow both need a 3-5 sentence plan stated first — don't generate first
  and explain after.
- **Never let a stub look finished.** If you hardcode a happy path, skip an edge case, or
  leave a "handle this properly later," say so explicitly in the summary and in a code
  comment.
- **STOP and wait for the developer's explicit go-ahead** at every point marked `STOP`
  below. These are the points where a subtly wrong implementation is easy to miss and hard
  to undo later (auth/token handling, file upload validation).
- **Write tests alongside each unit, not after, and state what they actually verify** — if
  a test only checks that a mock was called, say that plainly rather than implying it
  proves correctness.

## File uploads go directly to Cloudinary, not through the backend
Resume files (PDF/DOCX) are uploaded directly from the browser to Cloudinary using an
**unsigned upload preset** (a preset configured in the Cloudinary dashboard restricted to
`raw` resource type, `pdf`/`docx` formats, and a max file size — you don't need a Cloudinary
API secret in the frontend, ever). After a successful upload, Cloudinary returns a
`secure_url` and `public_id`; POST those two fields (plus original filename) to the
backend's `/resumes/` endpoint, which stores the reference and triggers analysis. Show
real upload progress using Cloudinary's upload API progress callback, not a fake spinner.

## Build in this order
1. Scaffold: Vite + React + TS + Tailwind, ESLint + Prettier, folder structure per Section 5.
2. API client layer: typed axios instance with interceptors for attaching the JWT access
   token and silently refreshing it on 401 (queue concurrent requests during refresh, don't
   fire N parallel refresh calls).
3. Auth pages: register, login, logout, protected-route wrapper that redirects to /login
   if there's no valid session, persisted auth state in a zustand store (memory + refresh
   via httpOnly cookie — never put the access token in localStorage).

   **STOP — review with the developer before continuing.** Auth and token storage are the
   easiest place for a plausible-looking mistake (e.g. a token in localStorage, a refresh
   race condition) to slip through unnoticed. Walk through the interceptor logic line by
   line with the developer before building anything on top of it.
4. Mock the API with MSW using the contract in Section 3, so every subsequent page can be
   built and tested without a live backend.
5. Dashboard shell: layout, nav, the page scaffolding for all 6 feature areas (resume,
   careers, jobs, interview, learning, chat).
6. Resume page: Cloudinary direct upload widget/dropzone, upload progress, polling or
   re-fetch for analysis status (uploaded → parsing → parsed → failed), render extracted
   skills/experience once parsed.

   **STOP — review with the developer before continuing.** Confirm together that file type
   and size are actually validated client-side (not just trusted to the Cloudinary preset),
   and that the upload preset is genuinely restricted to `raw`/`pdf`/`docx` before treating
   this as done.
7. Careers page: skill-gap visualization (radar or bar chart via recharts) and ranked
   career path cards with the AI's reasoning shown (not hidden behind a tooltip — this is
   the actual value of the product, surface it).
8. Learning page: roadmap as an ordered, checkable list/timeline with estimated hours.
9. Jobs page: ranked job match cards with fit score and the "why this matches" explanation
   prominently shown; save/dismiss actions.
10. Interview page: question list by category (behavioral/technical), inline answer
    textarea, submit for AI feedback, show score + feedback once returned.
11. Chat page: streaming chat UI consuming Server-Sent Events from
    `POST /chat/sessions/{id}/messages/`, rendering tokens as they arrive, with a visible
    "AI is busy, please retry" state for 429s (the AI backend runs on a free LLM tier with
    real rate limits — design for that, don't assume infinite throughput).
12. Replace MSW mocks with the real backend base URL once it's available; verify every flow
    end to end against the live API.
13. Tests: Vitest + React Testing Library per component/hook, Playwright for the 3 critical
    flows (login → upload resume → see analysis; chat round trip; interview Q&A).
14. Production build: containerize with Nginx per Section 9, verify `docker build` and a
    served `dist/` works with the production API base URL injected at build time.

## Definition of done
Every page in Section 7 works end-to-end against the mocked API with correct loading/empty/
error states, the Cloudinary direct-upload flow works against a real (free-tier) Cloudinary
account, auth (including silent token refresh) works, the chat page streams and handles
rate-limit errors gracefully, and `npm run build` produces a deployable static bundle that
runs correctly behind the Nginx config in Section 9. Just as important: the developer has
actually read and can explain the auth/token and upload-validation code, not merely
observed that tests pass for them.

Stop at every `STOP` checkpoint above and wait for explicit confirmation before continuing.
Outside of those checkpoints, ask only if something here contradicts itself — otherwise
make the standard, production-grade choice and leave a one-line comment explaining it.
````

## 2. Free-Tier Considerations That Affect the Frontend

- **Cloudinary free plan** is usage-capped (storage + bandwidth + transformations share one
  monthly pool). Don't request unnecessary transformations on resume files (they're
  documents, not images — store as `resource_type: raw`, don't run them through image
  transforms).
- **The AI backend runs on Gemini's free tier**, which has real per-minute and per-day
  request caps. The chat UI and every "Generate" button (skill gaps, roadmap, job matches,
  interview questions) must handle a `429`/`503` response with a clear, friendly "the AI is
  busy, try again in a moment" message and a disabled-with-cooldown retry button — not a
  silent failure or an infinite spinner.

## 3. API Contract (source of truth — identical in the backend prompt)

Base URL: `VITE_API_BASE_URL` (e.g. `https://api.yourdomain.com/api/v1`). All authenticated
requests send `Authorization: Bearer <access_token>`. Standard error shape:

```json
{ "error": { "code": "string", "message": "human readable", "details": {} } }
```

```
POST   /auth/register/                 { email, password, full_name } -> 201 { user, tokens }
POST   /auth/login/                    { email, password } -> 200 { user, tokens }
POST   /auth/refresh/                  (refresh cookie) -> 200 { access }
POST   /auth/logout/                   -> 204

GET    /me/                            -> 200 Profile
PATCH  /me/                            { current_role?, years_experience?, target_roles?, location?, bio? } -> 200 Profile

POST   /resumes/                       { cloudinary_url, cloudinary_public_id, original_filename } -> 201 Resume
GET    /resumes/                       -> 200 Resume[]
GET    /resumes/{id}/                  -> 200 Resume   # includes status: uploaded|parsing|parsed|failed
GET    /resumes/{id}/analysis/         -> 200 ResumeAnalysis | 404 if not parsed yet

POST   /careers/paths/generate/        { resume_id, target_role? } -> 202 { job_id }   # async
GET    /careers/paths/                 -> 200 CareerPath[]
GET    /careers/skill-gaps/{resume_id}/?target_role=... -> 200 SkillGap

POST   /jobs/matches/generate/         { resume_id } -> 202 { job_id }
GET    /jobs/matches/                  -> 200 JobMatch[]
PATCH  /jobs/matches/{id}/             { status: "saved"|"applied"|"dismissed" } -> 200 JobMatch

POST   /interviews/sessions/           { target_role } -> 201 InterviewSession
POST   /interviews/sessions/{id}/questions/generate/ -> 202 { job_id }
GET    /interviews/sessions/{id}/      -> 200 InterviewSession (with questions[])
POST   /interviews/questions/{id}/answer/ { user_answer } -> 200 { ai_feedback, score }

POST   /learning/roadmaps/generate/    { skill_gap_id } -> 202 { job_id }
GET    /learning/roadmaps/             -> 200 LearningRoadmap[]
PATCH  /learning/resources/{id}/       { completed: true } -> 200 LearningResource

POST   /chat/sessions/                 -> 201 ChatSession
GET    /chat/sessions/{id}/messages/   -> 200 ChatMessage[]
POST   /chat/sessions/{id}/messages/   { content } -> text/event-stream (SSE token stream)

GET    /jobs/async-status/{job_id}/    -> 200 { status: "pending"|"done"|"failed", result? }
```

`POST` endpoints marked `202` are async (they kick off an AI agent job); poll
`/jobs/async-status/{job_id}/` or treat the relevant `GET` list endpoint as eventually
consistent and re-fetch with React Query's `refetchInterval` while status is pending.

## 4. Cloudinary Direct Upload — Frontend Implementation Notes

```ts
// upload a resume file straight from the browser, unsigned
async function uploadResumeToCloudinary(file: File, onProgress: (pct: number) => void) {
  const formData = new FormData();
  formData.append("file", file);
  formData.append("upload_preset", import.meta.env.VITE_CLOUDINARY_UPLOAD_PRESET);
  formData.append("resource_type", "raw"); // documents, not images

  const url = `https://api.cloudinary.com/v1_1/${import.meta.env.VITE_CLOUDINARY_CLOUD_NAME}/raw/upload`;

  return new Promise<{ secure_url: string; public_id: string }>((resolve, reject) => {
    const xhr = new XMLHttpRequest();
    xhr.open("POST", url);
    xhr.upload.onprogress = (e) => onProgress(Math.round((e.loaded / e.total) * 100));
    xhr.onload = () => (xhr.status === 200 ? resolve(JSON.parse(xhr.responseText)) : reject(xhr.responseText));
    xhr.onerror = () => reject(new Error("Upload failed"));
    xhr.send(formData);
  });
}
```

Validate file type (`.pdf`, `.docx`) and size (≤5MB) client-side *before* this call, in
addition to whatever the Cloudinary upload preset itself restricts — never rely on a single
layer of validation.

## 7. Environment Variables (`.env.example`)

```
VITE_API_BASE_URL=https://api.yourdomain.com/api/v1
VITE_CLOUDINARY_CLOUD_NAME=your_cloud_name
VITE_CLOUDINARY_UPLOAD_PRESET=resume_unsigned_preset
```

All three are safe to expose in a client bundle — none is a secret. The Cloudinary API
secret stays server-side only (or, in this design, isn't needed at all since uploads are
unsigned and restricted via the preset's own rules in the Cloudinary dashboard).

## 8. Docker

```dockerfile
# frontend/Dockerfile
FROM node:20-slim AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
ARG VITE_API_BASE_URL
ARG VITE_CLOUDINARY_CLOUD_NAME
ARG VITE_CLOUDINARY_UPLOAD_PRESET
RUN npm run build

FROM nginx:1.27-alpine
COPY --from=build /app/dist /usr/share/nginx/html
COPY nginx.frontend.conf /etc/nginx/conf.d/default.conf
EXPOSE 80
```

```nginx
# frontend/nginx.frontend.conf
server {
    listen 80;
    root /usr/share/nginx/html;
    index index.html;
    location / {
        try_files $uri $uri/ /index.html;   # SPA fallback for react-router
    }
}
```

## 9. Testing

- **Unit/component:** Vitest + React Testing Library, MSW for any component that calls the
  API directly.
- **Contract tests:** keep `mocks/handlers.ts` in lockstep with Section 3 — if the backend
  team changes a response shape, this is the single file to update, and it should break
  visibly (failing tests) if it goes stale.
- **E2E:** Playwright against either the MSW-mocked build or the real backend, covering:
  register → login → upload resume → see parsed analysis; open chat → send message → see
  streamed response; generate interview questions → submit an answer → see feedback.

## 10. Definition of Done

`npm run build` produces a static bundle served correctly by the Nginx config above; every
page has working loading/empty/error states; auth (including silent refresh) works; resumes
upload directly to Cloudinary with real progress feedback; the chat page streams tokens and
degrades gracefully under rate limiting; Vitest and Playwright suites pass in CI. Equally
required: the developer stopped at each `STOP` checkpoint in Section 1, read the generated
auth and upload code, and can explain it — not just observed a green CI run.
